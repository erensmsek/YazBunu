# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A graduation project (BTK Bitirme Projesi): a fully local speech-to-text application. No cloud hosting, no Hugging Face Spaces, no ZeroGPU — everything runs on the user's MacBook Air M4 (16GB). Hugging Face Hub is used only as a model registry (downloading weights once, then running entirely offline/locally).

Stack:
- **Transcription**: `large-v3-turbo` Whisper, 8-bit quantized, via `mlx-whisper` (Apple Silicon-native, Metal-accelerated) — chosen over `whisper.cpp` and over PyTorch+MPS.
- **Translation**: `facebook/nllb-200-distilled-600M` via `transformers`, arbitrary source→target language (not just X→English like Whisper's built-in translate task).
- **Speaker diarization**: `pyannote/speaker-diarization-3.1` via `pyannote.audio` (gated model, currently blocked — see Current Status).
- **Summarize + polish (LLM)**: `Qwen/Qwen2.5-7B-Instruct` (BF16) via **Hugging Face Inference API** (`llm.py`). This is the one **deliberate exception** to the "everything local" rule — see gotcha #7. The local 4-bit MLX version was tested and rejected (too slow ~35s, hallucinations, degraded Turkish).
- **Backend**: FastAPI (`server.py`), single process, no build step.
- **Frontend**: hand-written HTML/CSS/JS in `static/` — deliberately **not** Gradio/Streamlit. Chosen for a more polished, custom-designed UI since the app never needs to be deployed to Spaces.

## Setup & running

```bash
cd "BTK Bitiriş Projesi"
source venv/bin/activate          # venv already created, deps already installed
export $(grep -v '^#' .env | xargs)   # loads HF_TOKEN (needed for gated diarization model)

HF_HUB_DISABLE_XET=1 uvicorn server:app --host 127.0.0.1 --port 7860
```

Open `http://127.0.0.1:7860`. There is no separate build/lint/test tooling in this repo — it's a small FastAPI app plus static assets, run directly.

To reinstall dependencies from scratch: `pip install -r requirements.txt` (see gotcha #2 below — `hf_xet` must stay uninstalled).

**Note on the Browser-pane preview tool**: `preview_start` with a `name` (launch.json config `stt-server`) fails with "Operation not permitted" in this project directory, likely because the path contains a space and a Turkish character. Workaround: start `uvicorn` manually in the background via Bash, then call `preview_start` with just a `url` pointing at the already-running `http://127.0.0.1:7860`.

## Architecture

`server.py` is the single entry point (FastAPI). It auto-detects `IS_APPLE_SILICON` at import and picks a provider per request via `_use_local(mode)` (see gotcha #8 for the dual-mode design). Routes:
- `GET /` — serves `static/index.html`.
- `GET /config` — `{apple_silicon, default_mode}`. The frontend uses this to decide whether to show the Lokal/API mode toggle (Apple Silicon only).
- `POST /transcribe` — multipart form: `file` (audio), optional `diarize` (bool), optional `mode` ("api"|"local"), optional `api_key` (Groq). Pipeline: transcribe (local `mlx_whisper` if `_use_local`, else `groq_api.transcribe`) → build segment list → if `diarize`, merge speaker labels from `diarize.py` (always local). Returns `{language, text, segments[]}` — **no translation here** (see New Features #8 UX rework below: translation moved to its own on-demand endpoint). Groq errors surface as HTTP 502 with a user-facing Turkish message.
- `POST /translate` — JSON body `{segments[], source_lang, target_lang, mode?, api_key?}` → `{target_lang, text, segments[]}`. Translates an *already-transcribed* transcript on demand — the frontend calls this only when the user picks a target language and clicks "Çevir" (see New Features #8), not automatically at transcribe time. Routes local NLLB (`translate.py`) vs Groq (`groq_api.translate_texts`) the same way `/transcribe` routes transcription.
- `POST /summarize` — JSON body `{text, api_key?, lang?}` → `{summary}`. `_run_llm` routes to `groq_api.summarize` when `api_key` is present, else falls back to `llm.summarize()` (HF Inference API). `lang` (ISO 639-1) makes the output language follow whatever the user has selected in the "Dil" picker — defaults to Turkish when omitted (see gotcha #10). See also gotchas #7 and #8.
- `POST /polish` — JSON body `{text, api_key?, lang?}` → `{polished}` (Markdown). Same routing and `lang` behavior as `/summarize`. The frontend renders the Markdown client-side (minimal renderer in `script.js`) and lets it be exported via `/export`.
- `POST /export` — JSON body `{format, text, segments[]}` → downloadable file via `export.py`'s `EXPORTERS` dict. Called for original, translated (if present), and polish outputs.
- `@app.on_event("startup")` — `ensure_model_weights_alias()`; now **guarded by `IS_APPLE_SILICON`** so the 824MB MLX model isn't downloaded on API-only platforms. See gotcha #1.

Supporting modules (each is a standalone, lazily-initialized wrapper — same pattern: a module-level `_model`/`_pipeline` singleton populated on first use):
- `export.py` — pure formatting functions, no I/O: `to_txt`, `to_srt`, `to_vtt`, `to_md`, `to_json`, all registered in `EXPORTERS: {format: (fn, media_type)}`.
- `translate.py` — NLLB wrapper. `LANG_MAP` (ISO 639-1 → FLORES-200 code) is the single source of truth for which languages the UI's translate dropdown should offer — extend both `LANG_MAP` here and the `<select id="targetLang">` in `static/index.html` together when adding a language. `translate_texts()` batches all segment texts into one `model.generate()` call rather than one call per segment. Note: `LANG_MAP` currently has 20 entries, but the `<select id="targetLang">` in `index.html` only exposes 12 of them (missing `nl`, `pl`, `sv`, `uk`, `el`, `hi`, `fa`, `ro`) — relevant when picking up New Features item 8 (language picker UI).
- `diarize.py` — pyannote wrapper, **always local** regardless of API/Lokal mode. Builds a `SpeakerDiarization` pipeline by hand from weights vendored in `models/diarization/` (see gotcha #8's diarization update) instead of downloading a pipeline from HF at runtime. Before running the pipeline, `diarize()` transcodes the input to a temp 16kHz mono WAV via `ffmpeg` (see gotcha #9 — required because pyannote/torchaudio can't read duration from webm/opus containers in this environment). `diarize()` returns raw speaker turns; `assign_speakers()` merges them onto Whisper segments by max-time-overlap and relabels pyannote's raw IDs (`SPEAKER_00`, ...) into sequential `Konuşmacı_1`, `Konuşmacı_2`, ... in order of first appearance (deliberately no gender/identity detection — sequential labels only, per project requirements).
- `groq_api.py` — **NOT local**: Groq (OpenAI-compatible) wrapper over plain `requests`, the single provider for **API mode**. `transcribe()` (Whisper via `/audio/transcriptions`, `verbose_json` → same dict shape as `mlx_whisper`), `translate_texts()` (batch translate through a chat LLM with a strict JSON-in/JSON-out prompt to preserve segment alignment — the API-mode replacement for NLLB), `summarize()`/`polish()` (same prompts/contract as `llm.py`). One Groq key covers all of these. Raises `GroqError` (user-facing Turkish messages) on 401/403/429/other. Model ids are module constants (`TRANSCRIBE_MODEL`, `LLM_MODEL`) — confirm against Groq's live catalog. See gotcha #8.
- `llm.py` — **NOT local** (unlike every other module): thin wrapper over `huggingface_hub.InferenceClient` calling `Qwen/Qwen2.5-7B-Instruct` via the `featherless-ai` provider (BF16, full precision). Same lazy-singleton pattern (`_client`). `summarize()` → short Turkish summary (narrow `max_tokens` + explicit "sadece Türkçe" to prevent the observed Chinese-drift glitch); `polish()` → Markdown-formatted rewrite with title. Both use a system+user chat template and `temperature=0.4`. Needs `HF_TOKEN` in `.env`. See gotcha #7 for why this is API-based, not local.

`static/` — no framework, no build step:
- `index.html` — top nav has a diarize `<checkbox>` shared by all input modes, a tab switcher (Mikrofon / Dosya Yükle / Sekme Dinle — see gotcha #12), and a results section with two parallel blocks (original transcript, optional translation). The target-language `<select id="targetLang">` and a "Çevir" button live in the *Transkript* block's header (not the nav — see gotcha #10), so language selection happens after a transcript exists and can be changed/re-triggered any time. Each block has its own segment list and export button row.
- `script.js` — `MediaRecorder`-based mic capture (see gotcha #4), drag-and-drop + `<input type=file>` upload, `getDisplayMedia`-based tab-audio capture (see gotcha #12), renders segments (with speaker-tag prefix when present) and drives `/export` downloads via `Blob` + a synthetic anchor click. Also: live **waveform** via Web Audio `AnalyserNode` drawn on a `<canvas>` — factored into a reusable `createWaveform(canvas)` closure (`{start(stream), stop()}`) since gotcha #12 added a second, independent recording surface that needed its own visualizer instance; `Özetle`/`Başlıklandır & İyileştir` buttons that POST the transcript to `/summarize` & `/polish`; and a **minimal Markdown renderer** (`renderMarkdown()` — handles `#`/`##`/`###`, `- `/`1.` lists, `**bold**`, `*italic*`, `` `code` ``) for the LLM output. No external Markdown lib (keeps the no-build-step constraint).
- `style.css` — CSS custom properties, `prefers-color-scheme` for dark/light, no framework. Refreshed to a modern rounded light theme (indigo accent, gradient-bordered AI cards, pill settings, animated record button + waveform). Light is the primary/designed-for theme; dark is a maintained fallback.

## Key technical decisions & gotchas

1. **Model weights filename mismatch (already fixed — don't reintroduce)**: `mlx-community/whisper-large-v3-turbo-8bit` ships weights as `model.safetensors`, but `mlx_whisper`'s `load_model()` only looks for `weights.safetensors` / `weights.npz`. `_ensure_mlx_weights()` in `server.py` hardlinks one name to the other (no data duplication, no re-download). **Since 2026-07-16 this runs lazily**, not on app startup — it's called from inside `_transcribe_local()`, guarded by a `_mlx_weights_ready` flag, so the 824MB download only happens the first time a request actually uses Lokal mode (see gotcha #8; this replaces the old startup-hook version that downloaded unconditionally on every boot on Apple Silicon, even for API-mode-only users — a real bug caught and fixed the same day it was introduced). If `MODEL_REPO` is ever pointed at a different `mlx-community` quantization, check its actual filenames first via `huggingface.co/api/models/<repo>` — some repos (e.g. `whisper-large-v3-turbo-q4`, the plain `whisper-large-v3-turbo`) already use `weights.safetensors`/`weights.npz` natively and need no fix; others (`-8bit`, `-4bit`, `-asr-*` variants) use `model.safetensors` and need this same alias trick.

2. **Hugging Face's "Xet" CDN (`cas-bridge.xethub.hf.co`) has been unreliable in this environment**: large public-model downloads have repeatedly stalled for many minutes at a fixed byte offset before either completing or requiring a retry. `hf_xet` (the accelerated Xet client) has been uninstalled to force the classic HTTP download path — keep it uninstalled (don't let `pip install -r requirements.txt` silently pull it back in via `transformers`/`huggingface_hub` extras). Also always export `HF_HUB_DISABLE_XET=1` as a second guard. If a download appears stuck, check whether `~/.cache/huggingface/hub/**/*.incomplete` is actually growing over ~30s before concluding it's broken.

3. **Gated diarization model — currently blocked, see Current Status.** `pyannote/speaker-diarization-3.1` requires accepting gated terms for *both* itself and its dependency `pyannote/segmentation-3.0` (two separate gates) at huggingface.co while logged in, plus an `HF_TOKEN` (read scope) in `.env`. Acceptance status is visible at `https://huggingface.co/settings/gated-repos`. Also note: `pyannote.audio`'s `Pipeline.from_pretrained()` takes a `token=` kwarg, not `use_auth_token=` (older API, removed in the installed version).

4. **Mic recording quality**: `script.js`'s `getUserMedia` call explicitly sets `echoCancellation`, `noiseSuppression`, and `autoGainControl` to `false`, and `MediaRecorder` is given `audioBitsPerSecond: 128000`. Browser defaults are tuned for VoIP calls and were measurably degrading Whisper's transcription accuracy (especially non-English) on real microphone input versus a clean uploaded file — don't revert this without re-testing accuracy.

5. **`mlx_whisper.transcribe()` caches its loaded model** as a class attribute (`ModelHolder`, keyed by the `path_or_hf_repo` string) across calls within the same process — as long as `MODEL_REPO` doesn't change between requests, the model loads once and stays warm. No additional caching layer needed in `server.py`.

6. **Everything in `/transcribe` runs synchronously inside an `async def` handler** (`mlx_whisper.transcribe`, `translate_texts`, `diarize` are all blocking calls) — the server can't service a second request while one transcription is in flight. Acceptable for the intended single-user local use; would need `run_in_executor`/threading if concurrent usage is ever required. Note `/summarize` and `/polish` are also blocking (synchronous HTTP call to HF) — same single-user assumption.

7. **Summarize/polish LLM is API-based, NOT local — a deliberate, user-approved exception to the "everything local" principle.** We benchmarked local MLX quantizations (`mlx-community/Qwen2-1.5B`, `Qwen2.5-3B`, `Qwen2.5-7B-Instruct-4bit`) against the API. The 1.5B/3B produced broken Turkish (invented words, wrong grammar, ignored instructions); the local 7B-4bit was usable but had hallucinations (invented "HearSay" label), broken numbering, minor grammar slips, and was **slow (~35s for a polish)**. The BF16 model over HF Inference API was clearly the cleanest and fast (~7-12s), so `llm.py` calls it remotely. Provider notes: `Qwen/Qwen2.5-7B-Instruct` is served by two providers for this token — `together` (FP8 "-Turbo", quantized) and `featherless-ai` (original BF16). We pinned `featherless-ai` for best polish quality. The MLX 4-bit repo canNOT be served via the API (Apple-Silicon-only format) — only local. The user's rationale for accepting cloud here: the project's grading is partly about HF-platform usage, and for personal use the only real blockers are cost/hardware, not the local-privacy purity that governs the other three models. If privacy ever matters again, the local 7B-4bit path already works (`test_llm.py`) — just slower/rougher. `test_llm_hf_api.py` is a companion scratch script for probing the HF Inference API directly (model/provider as CLI args, no server needed) — useful when re-tuning prompts for New Features item 3; `test_output_polish.md` is a saved sample `/polish` output from an earlier run, kept for before/after comparison.

8. **Dual-mode architecture (Lokal / API) with a Settings screen — IMPLEMENTED 2026-07-16.** Goal: anyone cloning the repo from GitHub, on any OS, can try the app in under a minute without downloading multi-GB weights or owning Apple Silicon, while the existing fully-local Mac path is kept as an option. **What shipped**: `groq_api.py` (Groq wrapper: `transcribe`/`translate_texts`/`summarize`/`polish`, all via one Groq key, plain `requests`); `server.py` refactored (Apple-Silicon auto-detect at load, `GET /config`, `mode`+`api_key` form fields on `/transcribe`, `api_key` in `/summarize`+`/polish`, mlx import made lazy + MLX weight download skipped on non-Apple so the app boots cross-platform); Settings modal + first-launch Groq banner in `static/` (mode toggle shown only on Apple Silicon, Groq key stored in `localStorage`, "Groq anahtarı al" links to `https://console.groq.com/keys`); `requirements.txt` gained a `platform_system=="Darwin" and platform_machine=="arm64"` marker on `mlx-whisper` so `pip install` no longer fails off-Apple; `.env.example` documents `GROQ_API_KEY` (UI is primary; env is a server-side fallback that `groq_api._require_key` also reads). **Routing model** (see `server.py`): transcription & translation follow the mode (local → `mlx-whisper`/NLLB, api → Groq); summarize & polish always go to Groq when a key is present, else fall back to the HF `llm.py` path (dev/back-compat); diarization is always local. **Bug found & fixed same day (2026-07-16)**: the initial implementation kept the old `@app.on_event("startup")` hook, which on Apple Silicon downloaded the 824MB mlx-whisper model **unconditionally on every boot**, regardless of whether the user ever picks Lokal mode — defeating the point of API mode being the lightweight default. Fixed by making the download lazy: `_ensure_mlx_weights()` is now called only from inside `_transcribe_local()`, gated by a `_mlx_weights_ready` flag, so it only fires the first time a Lokal-mode transcription actually runs. See gotcha #1.

**Verified** (2026-07-16, curl + in-browser): local transcription, local NLLB translation, API-mode-without-key friendly error, summarize HF-fallback, the full Settings UI (mode switch + key save → `localStorage`, banner/modal show-hide), and that Apple-Silicon startup no longer downloads anything until Lokal mode is actually used.

**Also verified live against Groq with a real key** (2026-07-16, curl reading `.env`'s `GROQ_API_KEY` via `export $(grep ...)` — the key value was never printed/echoed/logged in the process). All four Groq-backed paths return correct output end-to-end: `/transcribe` mode=api (Whisper), `/transcribe` mode=api+target_lang (chat-LLM translation, segment count/alignment preserved), `/summarize`, `/polish`. Two real bugs were caught and fixed during this pass — both live only in `groq_api.py`, nothing else needed to change:
   - **Language-code mismatch**: Groq's Whisper endpoint returns the detected language as a full name (`"Turkish"`), while `mlx_whisper` returns an ISO 639-1 code (`"tr"`). `server.py`'s `target_lang != language` check and the frontend's `LANG_NAMES[data.language]` lookup both expect ISO codes — with the raw Groq response, translation would silently misfire (always/never trigger) and the language pill would show the raw English name. Fixed with `groq_api._normalize_language()` (builds a name→code map from the existing `LANG_NAMES` dict, applied to `transcribe()`'s return value).
   - **`qwen/qwen3-32b` leaks its chain-of-thought into the response**: without an explicit setting, Groq returns the model's `<think>...</think>` reasoning inline in `message.content`, ahead of the actual answer — confirmed via a raw `/summarize` call whose output opened with a full English reasoning trace before the Turkish summary. Fixed by adding `"reasoning_effort": "none"` to every `_chat()` call in `groq_api.py` (Groq-documented parameter for Qwen3 models — disables reasoning entirely rather than just hiding it, which is also cheaper/faster for our use case since none of summarize/polish/translate need multi-step reasoning). Re-verified clean after the fix.

Both `TRANSCRIBE_MODEL="whisper-large-v3"` and `LLM_MODEL="qwen/qwen3-32b"` are confirmed live and working on Groq as of 2026-07-16. Original design decisions, still accurate:
   - **Transcription + translation + summarize/polish will all move to a single Groq API key in API mode** (Groq hosts both Whisper-large-v3 and chat LLMs, e.g. `qwen/qwen3.6-27b` — check current model id in Groq's console at implementation time since Groq deprecated `llama-3.3-70b-versatile`/`llama-3.1-8b-instant` in June 2026). One signup (no credit card), one key pasted into Settings, covers the whole API mode — this replaces the earlier idea of a separate HF key for the LLM. Groq free tier (verified 2026-07-16): Whisper ~2000 req/day & 7200 audio-sec/hour; chat models ~30 RPM, 6-30k TPM, up to 14,400 req/day — well above HF Inference API's free limits. OpenAI's Whisper API was considered and **rejected as the default**: no ongoing free tier, $0.006/min, only a one-time $5 credit that expires in 3 months.
   - **Diarization stays local-only, but its weights will be vendored directly into the repo** instead of downloaded from HF at runtime: `pyannote/segmentation-3.0` (~5.9 MB, MIT) + `pyannote/wespeaker-voxceleb-resnet34-LM` (~26.6 MB, CC-BY-4.0) — both license-cleared for redistribution, total ~32 MB (tiny next to Whisper's 824 MB or NLLB's 2.3 GB). This also permanently sidesteps the Xet CDN 403 issue in the "Current status" section above, since the gated `pyannote/speaker-diarization-3.1` pipeline download is never invoked — instead build the same `pyannote.audio.pipelines.SpeakerDiarization` pipeline in code pointing at the local weight files, with the upstream pipeline's exact hyperparameters hardcoded (pulled from its `config.yaml` on 2026-07-16): `clustering: AgglomerativeClustering`, `method: centroid`, `threshold: 0.7045654963945799`, `min_cluster_size: 12`, `embedding_batch_size`/`segmentation_batch_size: 32`. Add a small attribution/citation note (courtesy, not license-required) crediting the pyannote project.
   - **Mac users can still choose Lokal mode** (existing `mlx-whisper` + local NLLB + local diarization, unchanged) via the new Settings screen.
   - **Settings screen** (new, in `static/`): mode toggle (Lokal / API), Groq API key field, a short "buradan anahtar al" instruction/link, key stored client-side (e.g. `localStorage`) or in the user's own `.env` — never committed, ties into New Features item 11.
   - **Groq's Whisper endpoint has its own file-size cap (~25-30 MB on the free tier, verify exact number in Groq's console at implementation time) — separate from the model's internal 30-second processing window** (Whisper's encoder always chunks audio into 30s windows internally; this happens automatically server-side on Groq exactly like it does inside local `mlx-whisper` — no client-side pre-chunking needed for it). At this app's typical recording size (mic capture is 128kbps opus per gotcha #4 — a 5-minute recording is only a few MB), the file-size cap is not a practical concern. It only becomes one for long uploaded files (raw/uncompressed WAV, or 10+ minute recordings). Two follow-ups noted for later, not yet built:
     - Add a client-side file-size warning to the upload dropzone in `index.html` (e.g. warn if an uploaded file is large enough to risk hitting the API-mode size cap).
     - For large uploads, transcode/compress server-side before sending to Groq (standard tool: `ffmpeg`, e.g. downmix to mono + re-encode at a modest bitrate) rather than rejecting the file outright — only trigger this when a file is close to/over the cap, to avoid re-encoding every small file unnecessarily.
   - **End-user account requirements, once this is fully built**: Lokal mode needs **no account at all** (Whisper/NLLB are public, non-gated HF repos — no token needed to download them; diarization weights are vendored in the repo, not downloaded). API mode needs **only a Groq account** (covers transcription + translation + summarize/polish under one key). `HF_TOKEN` stops being required for end users entirely once `llm.py`'s calls also move to Groq and diarization is vendored — it becomes a dev-only detail, if kept at all.
   - **Settings UX finalized (2026-07-16)**: no manual "are you on a Mac?" question — `server.py` auto-detects Apple Silicon at startup (`platform.machine()`), and the Settings screen conditionally shows a Lokal/API toggle only when Apple Silicon is detected; on any other platform the toggle is hidden entirely and API mode is used silently (no dead/confusing option shown). Diarization is unaffected by this toggle either way (see above — always local, cross-platform).
   - **Built**: the first-launch "Anahtar al" banner (links to Groq signup/keys) + "Anahtarı gir" (opens Settings), and the same `https://console.groq.com/keys` link inside the Settings modal next to the key field. Banner auto-hides once a key is stored.
   - **Module naming as built**: a single `groq_api.py` (not separate `transcribe_api.py`/`translate_api.py`); `server.py` picks the provider per request via `_use_local(mode)` (True only when Apple Silicon AND mode=="local"). Settings UI lives in the existing `static/` files (no new page).

9. **Diarization on webm/opus input 502'd until fixed (2026-07-16) — real bug, found via user report, not a synthetic-audio artifact.** Enabling "Konuşmacı ayrımı" in the browser (mic recording → webm/opus, per gotcha #4) always failed with `502` / `Diyarizasyon hatası: unsupported operand type(s) for *: 'NoneType' and 'int'`. Root cause: `pyannote.audio.core.io.Audio.crop()` calls `get_num_samples(metadata.duration_seconds_from_header, sample_rate)`, and this environment's `torchaudio` returns `None` for `duration_seconds_from_header` on webm/opus containers (it can't read duration from that container format's header) — `None * sample_rate` then crashes. `torchaudio.load()`/`.info()` aren't usable workarounds either — this env's torchaudio has also dropped `.info()` entirely (same underlying torchaudio-version fragility as the pyannote-3.x incompatibility in the "Diarization update" note above). **Fix**: `diarize()` in `diarize.py` now transcodes the input to a temp 16kHz mono WAV via `ffmpeg` *before* handing it to the pipeline — sidesteps pyannote's header-duration path entirely and also normalizes every input format (webm, mp3, m4a, wav, ...) through one path. **Requires `ffmpeg` on PATH** (a system binary, not a pip package — already assumed available in this project for other ad-hoc audio prep, but now a hard runtime dependency for diarization specifically; not yet added to a formal "system requirements" doc since none exists yet, worth remembering for New Features #11/#13). Verified fixed via curl with both `mode=local` and `mode=api`, using a synthetic webm/opus 2-speaker clip — both return `200` with `speaker` labels populated (the clustering-can't-tell-`say`-voices-apart limitation from gotcha #8 is separate and still open).

10. **Translation UX reworked from "pick language before recording" to "translate on demand after transcript exists" (2026-07-16) — real bug, found via user report.** Original design had a single "Çeviri" language `<select>` in the top nav, read only at the moment `/transcribe` fired (`sendForTranscription` appended `target_lang` to the form data). This meant selecting a language *after* a transcript was already on screen did nothing — no listener re-triggered translation, so users reported "I pick a language but it doesn't translate." Also, summarize/polish always hardcoded Turkish output regardless of any language selection, which wasn't what a non-Turkish-first user would expect. **Fix**: (1) `/transcribe` no longer accepts `target_lang` at all — it only transcribes (+ diarizes). (2) The language `<select>` moved from the top nav into the Transkript result block's header, next to a new "Çevir" button; clicking it POSTs the *current* transcript's segments to the new `/translate` endpoint, so translation is explicitly user-triggered after the transcript exists, any number of times, to any language, without re-transcribing. (3) `/summarize` and `/polish` gained an optional `lang` field — both `groq_api.py` and `llm.py`'s `summarize()`/`polish()` now take a `lang: Optional[str]` parameter and interpolate the target language name into the system prompt (`_lang_name()` helper in both modules, defaulting to `"Turkish"` when `lang` is falsy — preserves old behavior for anyone not using the picker). The frontend's `callLLM()` passes `targetLangSelect.value || null` as `lang`, so if the user has a target language selected, the AI summary/polish comes out in that language too — matching the user's actual request ("seçtiğim dilde özet ya da başlıklı içerik üretsin"). 13. **Local history ("Geçmiş") with append-to-existing-recording — IMPLEMENTED (2026-07-17), see New Features #6.** Every completed transcription is auto-saved to `localStorage` (key `yazbunuHistory`, capped at 50 records, oldest dropped first) — no server-side storage, nothing added to `server.py`; this is entirely a `script.js` feature (`History` object: `list/save/get/upsert/remove`).
